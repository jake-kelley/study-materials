Part 2 — Thorough Study Guide

1 · Exam Details

  • 57 questions in 60 minutes (~63 seconds each). Feels tight on scenario items — flag-and-return.
  • Four question types: multiple choice, multiple answer, true/false, and text-match / fill-in-the-blank (type the exact command or value — so know CLI spelling).
  • Passing score ~70% (~40/57); HashiCorp doesn't publish the exact cut score.
  • $70.50, online webcam-proctored, English only, results near-instant by email. Cert valid 2 years.
  • No hard prerequisite. Recommended: basic terminal skills, cloud familiarity, and hands-on Terraform.
  • Tested against Terraform 1.12 — questions reflect current CLI behavior, HCL features (custom conditions, dynamic expressions), and HCP Terraform.

2 · IaC Concepts & Why Terraform

  • Infrastructure as Code (IaC) = define infrastructure in declarative config files, version-controlled and repeatable. Benefits: consistency, repeatability, versioning, automation, self-documentation, reduced human error.
  • Declarative (Terraform, describe desired end state) vs imperative (scripts, describe steps).
  • Why Terraform specifically: cloud/provider-agnostic (one workflow across AWS/Azure/GCP + hundreds of providers), a huge provider ecosystem, and state that maps config → real resources so it can plan precise changes.
  • Purpose of state: track real-world resources, store metadata, improve performance, and enable collaboration.

Terraform vs the alternatives (a stated objective)

ToolHow Terraform differs
CloudFormation / ARM-BicepVendor-locked to one cloud; Terraform runs the same workflow across any provider — the answer to "multi-cloud IaC."
Ansible / Chef / PuppetPrimarily configuration management (software on existing machines, often procedural); Terraform is provisioning (create/modify/destroy the infrastructure itself, declarative). They complement: Terraform builds the VM, Ansible configures it.

3 · Terraform Basics & Providers

  • Plugin-based architecture: Terraform Core talks to providers (plugins) via RPC. Providers implement resources/data sources for a platform.
  • Declaring providers (in a terraform block):
    terraform { required_providers { aws = { source = "hashicorp/aws", version = "~> 5.0" } } }
  • How Terraform finds providers: from the public Terraform Registry by default (namespace/type), downloaded during terraform init into .terraform/.
  • Version constraints: =, !=, >=, <=, and the pessimistic ~>: ~> 5.1 allows 5.x ≥ 5.1 (not 6.0); ~> 5.1.2 allows 5.1.x only. required_version pins the Terraform CLI.
  • .terraform.lock.hcl (dependency lock file) records the exact provider versions selected — commit it.
  • You can use multiple providers (and multiple configurations of one provider via alias — e.g. two AWS regions; select with provider = aws.west on a resource).
  • Provider authentication: never hard-code credentials in config. Providers read environment variables (e.g. AWS_ACCESS_KEY_ID), shared credential files/profiles, or ambient identity (instance roles, OIDC). Hard-coded keys end up in VCS — the "which is the secure approach?" answer is always env vars / dynamic credentials.

4 · Core Workflow & CLI

The core workflow is Write → Plan → Apply. Know the exact behavior of each command — the exam tests it precisely.

CommandWhat it does (and gotchas)
terraform initInitializes the working dir: downloads providers & modules, configures the backend. Flags: -upgrade, -reconfigure, -migrate-state, -backend=false. Run it first / after adding providers or modules.
terraform planDry run — shows what will change. -out=planfile saves a plan; -target limits scope; -refresh-only previews drift.
terraform applyProvisions. apply planfile applies a saved plan without prompting. -auto-approve skips the yes/no. -replace=ADDR recreates a resource.
terraform destroyTears everything down (alias for apply -destroy). -target for partial.
terraform fmtRewrites config to canonical style. -check, -recursive, -diff. Only formatting.
terraform validateChecks syntax/internal consistency only — does NOT contact providers/APIs and needs no credentials. Requires init first.
terraform consoleInteractive REPL to test expressions, functions, and interpolations against state.
terraform output / show / graph / versionRead outputs; human/JSON view of state or a plan; dependency graph; version.
  • Verbose logging: set TF_LOG=TRACE|DEBUG|INFO|WARN|ERROR (TRACE is the most verbose; and TF_LOG_PATH to write to a file) for debugging.
  • taint/untaint mark a resource for recreation but are deprecated — use apply -replace=ADDR instead (know both).
  • Plan symbols: + create · - destroy · ~ update in place · -/+ destroy then recreate (a "forces replacement" attribute changed). Reading a plan snippet is a common question format.
  • CLI workspaces: terraform workspace new/select/list/show/delete — multiple state files for one configuration (e.g. dev/stage); interpolate the current one with terraform.workspace. Every config starts in the default workspace, which can't be deleted.

5 · State

  • Default backend is local (terraform.tfstate on disk). Remote backends (S3, azurerm, gcs, consul, pg, kubernetes, HCP Terraform via the cloud block) enable team collaboration.
  • Backend syntax: declared inside the terraform block — terraform { backend "s3" { bucket = "…", key = "…", region = "…" } }. Backend config can't use variables/interpolation; missing values can be supplied at init time (partial configuration via -backend-config). Changing backends requires re-running init (-migrate-state to carry state over).
  • State locking prevents concurrent writes from corrupting state. Remote backends support it (e.g. S3 historically used a DynamoDB lock table; HCP Terraform locks automatically). -lock=false and terraform force-unlock exist for edge cases.
  • Secrets in state: state stores values in plaintext, including secrets. Never commit state to VCS; use an encrypted remote backend and restrict access. sensitive = true only hides values in CLI output — they're still in state.
  • terraform_remote_state data source reads outputs from another config's state (cross-config wiring).
  • Drift = real infra differs from state; detect with plan / apply -refresh-only.

State-manipulation commands (top exam trap — memorize the semantics)

CommandEffect
state listList resources tracked in state.
state show ADDRShow attributes of one resource.
state mvRename/move a resource in state — refactor without destroy/recreate.
state rm ADDRRemove from state; the real resource is NOT destroyed — Terraform just "forgets" it.
import ADDR IDBring existing infra into state. The CLI form does NOT generate configuration — you must write the resource block yourself.

Config-driven import & refactoring (the modern way)

  • Declare an import { to = aws_x.this, id = "…" } block in your configuration, then run terraform plan / apply. Unlike the CLI command, this is reviewable in a plan and version-controlled.
  • terraform plan -generate-config-out=generated.tf will scaffold the resource configuration for imported resources so you don't hand-write it.
  • moved { from = aws_x.old, to = aws_x.new } blocks are the config-driven equivalent of state mv — rename/move resources (including into modules) without destroy/recreate, reviewable in the plan.
  • removed blocks are the config-driven state rm — stop managing a resource without destroying it.

6 · Modules

  • A module is a reusable container of resources. The top level is the root module; it calls child modules.
  • Sources: local paths (./modules/x), the Terraform Registry (namespace/name/provider), Git/GitHub (git::https://…, ?ref=v1.2.0), Bitbucket, S3/GCS buckets, HTTP.
  • Versioning: version = "~> 3.0" works for registry modules; for Git, pin with ?ref=<tag>. Local paths don't take a version.
  • Inputs = input variables (variable blocks); outputs = output blocks that expose values to the parent. Variables are scoped to their module — there are no global variables.
  • Calling a module: module "vpc" { source = "terraform-aws-modules/vpc/aws", version = "~> 5.0", cidr = "10.0.0.0/16" } — arguments map to the module's input variables. Read its outputs as module.vpc.vpc_id. After adding/changing a module, run terraform init (or get) to download it.
  • A module can only see what's passed in and only exposes what it outputs — parents can't reach arbitrary resources inside a child. That encapsulation is the tested concept behind "variable scope within modules."
  • Registry modules marked verified are reviewed by HashiCorp; the registry documents inputs/outputs and gives the version constraint to copy.

7 · Configuration & HCL

Variables & precedence (low → high)

  1. Environment variables (TF_VAR_name)
  2. terraform.tfvars
  3. terraform.tfvars.json
  4. *.auto.tfvars / *.auto.tfvars.json (alphabetical)
  5. -var and -var-file on the CLI (highest; last one wins)

Variable types. Primitives (string, number, bool); complex types — collection (list, set, map) and structural (object, tuple). Modifiers: default, sensitive, nullable, validation. A variable with no default and no supplied value → Terraform prompts interactively.

Named values — how you reference things

  • var.name = input variable · local.name = local value · module.name.output = child-module output · type.name.attr = resource attribute · data.type.name.attr = data source.
  • locals define named expressions computed once and reused (locals { common_tags = { env = var.env } } → local.common_tags) — DRY for repeated expressions; unlike variables, they can't be set from outside.
  • Outputs: output "ip" { value = aws_instance.web.public_ip } — shown after apply, read with terraform output, and the interface between modules. sensitive = true redacts it in CLI output.

Resources, data sources & references

  • resource "type" "name" {} manages infra; data "type" "name" {} is a read-only lookup refreshed during plan.
  • Cross-resource references connect resources: refer to another resource's attribute as type.name.attribute (e.g. aws_vpc.main.id).
  • Implicit dependencies form automatically when one resource references another; add an explicit depends_on only when there's no reference. Terraform builds a dependency graph (DAG) and parallelizes independent resources.

Dynamic configuration — expressions & functions

  • Built-in functions transform values: string (join, split, format, replace), collection (length, lookup, merge, concat, flatten), numeric, encoding (jsonencode), and more. Test them in terraform console.
  • Conditional expression: condition ? true_val : false_val.
  • for expressions transform lists/maps: [for s in var.list : upper(s)].
  • dynamic blocks generate repeated nested blocks from a collection (e.g. many ingress rules) instead of copy-paste.
  • Splat expressions (aws_x.this[*].id) collect an attribute across all instances.

count vs for_each

countfor_each
Indexed: aws_x.this[0]Keyed over a map/set: aws_x.this["key"]
Good for N identical copiesPreferred when items are distinguishable
Removing a middle item reindexes → churnStable keys → no reindex churn (each.key, each.value)

lifecycle meta-arguments

  • create_before_destroy = true — build replacement before destroying the old one.
  • prevent_destroy = true — hard-block deletion (plan errors out).
  • ignore_changes = [attr] (or all) — ignore drift on specific attributes.
  • Also meta-args: depends_on, count, for_each, provider.

Custom conditions & validation

  • Variable validation: a validation { condition = … , error_message = … } block inside a variable rejects bad input early (e.g. enforce an instance-type prefix or a CIDR shape).
  • Preconditions & postconditions: a lifecycle { precondition {…} / postcondition {…} } block on a resource/data source asserts assumptions before and guarantees after apply — failing the plan/apply with your message if violated.
  • check blocks define scoped assertions (with optional data sources) that report warnings without blocking apply — good for continuous validation of a deployment.

Managing sensitive data

  • Mark inputs/outputs sensitive = true to keep them out of CLI output — but remember they are still stored in state, so protect state itself.
  • Don't hard-code secrets in .tf files or commit .tfvars with secrets. Inject them at runtime via environment variables (TF_VAR_*) or a secrets manager.
  • Best practice is to pull secrets from a dedicated store — e.g. the Vault provider / data sources — rather than storing them in configuration. Ephemeral values/resources let sensitive data flow through a run without being persisted to state.

Provisioners — last resort

  • local-exec, remote-exec, file. HashiCorp says provisioners are a last resort — prefer native provider resources. on_failure = continue|fail; creation-time vs destroy-time. Expect a "which is best practice?" question favoring native resources.

8 · HCP Terraform (formerly Terraform Cloud)

  • Remote runs (plan/apply executed in HCP) and remote state storage with automatic locking & encryption.
  • VCS integration (GitHub/GitLab/Bitbucket) → speculative plans on PRs, run triggers.
  • Private module & provider registry.
  • Workspaces (org-level — different from CLI workspaces) and projects.
  • Sentinel policy-as-code (and OPA) for governance; cost estimation; run tasks; team/SSO/RBAC; variable sets.
  • Editions: Free / Standard / Plus. The exam tests which features are Cloud/Enterprise-only (Sentinel, cost estimation, run triggers, SSO, remote state sharing).
CLI workspaces ≠ HCP workspaces terraform workspace (CLI) manages multiple state files for one config, referenced via terraform.workspace. HCP workspaces are org-level managed environments. Don't confuse them.

11 · Tips & Common Traps

  • State semantics (top trap): state rm forgets but doesn't destroy; import adds to state but doesn't write config; state mv refactors without destroy/recreate.
  • count vs for_each — reindex churn with count; stable keys with for_each.
  • Version constraint ~> behavior and the lock file.
  • Cloud/Enterprise-only features — Sentinel, cost estimation, run triggers, remote state sharing, SSO. Know the boundary vs Community edition.
  • Provisioners are a last resort — favor native provider resources.
  • Exact CLI behavior: validate needs no credentials/API; apply savedplan doesn't prompt; fmt only reformats; init downloads providers/modules.
  • Fill-in-the-blank questions mean you must know exact command spelling — practice typing them.
  • Pace: 57 Q / 60 min ≈ 63 s each. Flag-and-return on scenarios.
Ignore Exam dumps (ExamTopics "actual questions", etc.) — they violate HashiCorp's certification agreement and risk decertification, are often outdated, and are unnecessary given Krausen/ExamPro. Also make sure any course you pick is current (aligned to the latest objectives and Terraform 1.12 — avoid stale 2021/2022 titles).
